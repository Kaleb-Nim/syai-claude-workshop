  <how-to-verify>
1. **Dev mode — copy button:**
   - Run `bunx slidev` (opens http://localhost:3030).
   - Navigate to slide 2 ("Code blocks work").
   - Hover the code block. Expect: a "Copy" button appears top-right of the code panel, styled in the deck's monospace + dim cream.
   - Click "Copy". Expect: label flips to "Copied" with rust accent color; reverts to "Copy" within ~2s.
   - Paste into any text editor. Expect: the exact unhighlighted source (3 lines starting with `import { query }`) — no extra whitespace before or after.

2. **Dev mode — sidebar:**
   - On any audience slide (e.g. slide 1), expect a vertical strip on the right edge:
     - Top line: `Slide 1 / 3` (or whatever total currently is).
     - Below: at minimum a "Start" entry; "Deep-linking works" appears as an entry on slide 3 (which has `layout: section`).
   - Navigate slides via arrow keys or the navbar — confirm the slide count increments and the active-section highlight (rust) updates to the most-recent section.
   - Click a section entry in the sidebar — confirm the deck jumps to that slide.

3. **Presenter mode:**
   - Open http://localhost:3030/presenter/ in a new tab.
   - Expect: standard Slidev presenter chrome (current slide left, notes/next-slide right, navbar at bottom).
   - Expect: NO `.deck-sidebar` strip overlaying the presenter UI.
   - Expect: NO copy buttons in the presenter's slide thumbnails or the next-slide preview pane.
   - Press right-arrow several times. Expect: slide changes propagate without layout twitch.

4. **Overview mode:**
   - Open http://localhost:3030/overview.
   - Expect: standard Slidev grid of all slides, NO sidebar overlay.

5. **Built preview:**
   - Stop the dev server. Run `bun run preview` (this builds and serves on port 4173).
   - Repeat steps 1–4 against http://localhost:4173. Confirm the copy button and sidebar behave identically to dev mode (this is the hard COMP-01 + COMP-05 requirement: works in built output).

6. **Console cleanliness:**
   - In each of dev and preview, open DevTools. Expect: no red errors. A single `[CopyButton] clipboard write failed` warning is acceptable ONLY if you tested in a context where clipboard permission was denied (insecure origin, etc.) — otherwise it should not appear.

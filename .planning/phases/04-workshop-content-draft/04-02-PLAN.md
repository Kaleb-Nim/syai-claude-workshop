**Slide 2 — Why care (D-08 step 2)**

# Why hooks

- **Guardrails.** Block tool calls that touch what you don't want touched.
- **Telemetry.** Log every tool call, every prompt, every session start.
- **Glue.** Run a script when Claude finishes a turn — re-run tests, format, deploy.

Hooks are how you make Claude Code react to itself.
```

**Slide 3 — Mental model: lifecycle events (D-08 step 3, D-18)**
List the five events per D-18 in a single slide. No v-clicks (D-12) — this is the reference card.
```markdown
# Lifecycle events

| Event | Fires when |
|-------|-----------|
| `SessionStart` | a new Claude Code session begins |
| `UserPromptSubmit` | you press enter on a prompt |
| `PreToolUse` | before Claude executes a tool call |
| `PostToolUse` | after a tool call completes |
| `Stop` | Claude finishes a turn |

Each hook is a shell command. Output and exit code matter — block by exiting non-zero from `PreToolUse`.
```

**Slide 4 — Mental model: settings.json shape (D-18)**
Show the canonical structure. Any field name you're unsure of: mark `<!-- VERIFY -->`.
```markdown
# settings.json shape

Hooks live in `~/.claude/settings.json` (user-wide) or `.claude/settings.json` (project).

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "*",
        "hooks": [
          { "type": "command", "command": "/path/to/script.sh" }
        ]
      }
    ]
  }
}
```

`matcher` filters by tool name (e.g. `"Edit"`, `"Bash"`, `"*"` for all). `command` runs in your shell with hook context piped on stdin.
```

**Slide 5 — Live walkthrough: prepare (D-08 step 4)**
```bash
# We'll wire a PostToolUse hook that logs every tool call.
# Step 1: create the log file and the script.
touch ~/.claude/hook-log.txt
mkdir -p ~/.claude/hooks

cat > ~/.claude/hooks/log-tool.sh <<'EOF'
#!/usr/bin/env bash
# Hook context arrives on stdin as JSON. We just append a timestamp
# and the raw payload to a log file.
ts=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
payload=$(cat)
echo "$ts $payload" >> ~/.claude/hook-log.txt
EOF

chmod +x ~/.claude/hooks/log-tool.sh
```

**Slide 8 — Mental model recap: the data flow**
```markdown
# What just happened

1. You sent a prompt → `UserPromptSubmit` fired (we didn't hook it).
2. Claude decided to call `Bash` → `PreToolUse` fired (we didn't hook it).
3. Bash ran → `PostToolUse` fired → our script appended one line to the log.
4. Claude's turn ended → `Stop` fired.

Five hookable events per turn. We tapped one.
```

**Slide 9 — Defensive hook (concept only, D-18)**
```markdown
# Defensive hook: don't show, deny

Block edits to anything matching `.env*` — exit non-zero from `PreToolUse`.

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "~/.claude/hooks/deny-env.sh" }
        ]
      }
    ]
  }
}
```

`deny-env.sh` reads the tool input on stdin, checks `file_path` against `.env`, and exits `1` to abort. We won't wire this live — easy to lock yourself out. <!-- VERIFY: exact exit-code semantics for PreToolUse deny -->
```

**Slide 10 — Debugging (D-18)**
```markdown
# When hooks don't fire

- **Did you start a new session?** Hook config loads at session start.
- **Is the script executable?** `chmod +x` matters.
- **Is the matcher right?** `*` for all tools; tool name otherwise.
- **Where is its output going?** Hooks run silently. Add `>> /tmp/hook-debug 2>&1` while debugging.
- **Path expansion?** `~` may not expand in JSON — use `$HOME` resolved at script time, or full absolute paths.
```
